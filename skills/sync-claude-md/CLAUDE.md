# Personal Preferences

## Tech Stack Preferences
When uncertain, prefer: Python

## Coding Standards
- Write self-documenting code. Clear names and structure should carry the explanation — not comments.
- Add a comment only when the *why* can't be expressed in the code itself (a tradeoff, a workaround, an external constraint). Skip comments that just restate what the code already says.
- Use full, descriptive names, not single letters or unexplained abbreviations — including for loop counters and short-lived variables.
- Prefer explicit over implicit, even when it costs a few extra lines.
- Follow the standard style guide for the language in use (e.g. PEP 8 for Python) unless the project already has its own established conventions.
- If a problem can be solved in a simpler way, propose it.
- Before adding or using a package, verify its latest stable version. If that stable version is years behind the current edge/latest release, flag this before using it so a decision can be made.
- Avoid adding dependencies unnecessarily. A new package is justified when it handles genuinely complex functionality — but if the functionality would be simpler to implement directly, or the package is extremely out of date, write it yourself instead.

## General preferences
- If asked to do too much work at once, stop and state that clearly.

## Research & Sourcing
- Prefer first-party/primary sources. If a claim rests on a secondary or tertiary source, verify it against the original before relying on it.

## Working Style
- If you don't know something, or you're acting on an assumption, say so explicitly at the point it matters — don't silently proceed, and don't wait to be asked. Flag it so I can steer early, rather than losing hours down a path that was never going to work.
